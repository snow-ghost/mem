# Спецификация: Подагент-компаньон памяти для Claude Code

## Цель

Реализовать систему персистентной памяти для ИИ-агента (Claude Code), основанную на архитектуре «подагент-компаньон + файловое хранилище». Система должна автоматически накапливать, консолидировать и предоставлять три типа памяти: эпизодическую (что было), семантическую (какие правила извлечены) и процедурную (проверенные рецепты).

## Обоснование выбора архитектуры

Подагент-компаньон выбран вместо MemGPT/Letta по трём причинам:

1. **Изоляция ошибок** — ошибки памяти не влияют на основную работу агента (37% отказов в мультиагентных системах = misalignment состояний)
2. **Нет lock-in** — работает поверх любого агента, не заменяя стек
3. **Прозрачность** — все данные в файлах, инспектируемых человеком

## Структура хранилища

```
project/
├── CLAUDE.md                        # Семантическая память (правила проекта, стек, границы)
│                                     # Ограничение: ≤150 строк, конкретные верифицируемые инструкции
│
├── .memory/
│   ├── episodes.jsonl                # Эпизодическая память (append-only)
│   ├── principles.md                 # Семантическая память (извлечённые правила)
│   ├── skills/                       # Процедурная память (проверенные рецепты)
│   │   └── {slug}.md                 # Один файл = один навык
│   └── consolidation-log.md          # Лог всех операций консолидации
```

### episodes.jsonl

Append-only лог значимых событий. Каждая строка — самостоятельная JSON-запись:

```json
{"ts": "2026-03-19T14:32:00Z", "session": "abc123", "type": "решение", "summary": "Выбрали JSONL вместо SQLite для эпизодов — проще диффить в Git", "tags": ["архитектура", "хранилище"]}
{"ts": "2026-03-19T15:10:00Z", "session": "abc123", "type": "ошибка", "summary": "Тесты падали из-за race condition в параллельной записи — добавили file lock", "tags": ["баг", "конкурентность"]}
{"ts": "2026-03-19T16:45:00Z", "session": "abc123", "type": "паттерн", "summary": "Третий раз за неделю ручная миграция БД — кандидат на навык", "tags": ["паттерн", "миграция"]}
```

**Типы событий:** `решение`, `ошибка`, `паттерн`, `инсайт`, `откат`

**Правила записи:**
- Записывать ТОЛЬКО значимые события, не каждый tool call
- `summary` — одно предложение, понятное без контекста сессии
- `tags` — 1–3 ключевых слова для группировки при консолидации

### principles.md

Извлечённые правила. Формат:

```markdown
# Принципы проекта

## Архитектура
- Для append-only логов использовать JSONL, не SQLite — проще диффить в Git, нет зависимости на драйвер
- Файлы памяти ≤150 строк — длиннее агент начинает игнорировать

## Тестирование
- При параллельной записи в файлы всегда добавлять file lock — иначе race condition
- Интеграционные тесты запускать с флагом -race

## Миграции
- Перед миграцией всегда делать snapshot текущей схемы
```

**Правила обновления:**
- Каждый принцип — одна строка, начинается с «-»
- Группировка по темам через заголовки `##`
- Максимум 100 строк — при превышении консолидировать (объединить похожие, удалить устаревшие)
- Каждый принцип должен быть проверяемым действием, не пожеланием

### skills/{slug}.md

Один навык — один файл. Формат:

```markdown
# Миграция базы данных

## Когда применять
- Нужно изменить схему БД в проекте

## Предусловия
- Доступ к БД
- Бэкап текущей схемы

## Шаги
1. Создать миграцию: `make migrate-create name=описание`
2. Написать SQL в созданном файле (формат Goose: +goose Up / +goose Down)
3. Применить: `make migrate-up`
4. Проверить статус: `make migrate-status`
5. Перегенерировать модели: `make model`
6. Обновить репозитории/маппинг если изменились столбцы

## Проверка успеха
- `make migrate-status` показывает Applied для новой миграции
- `make test` проходит
- `make lint` проходит

## Антипаттерны
- Не редактировать файлы в `internal/db/postgres/model/` — они генерируемые
- Не забыть секцию +goose Down для отката
```

**Критерий сохранения навыка:** действие выполнено 3+ раз ИЛИ единожды, но сложное и критичное.

### consolidation-log.md

Лог операций подагента-памяти:

```markdown
## 2026-03-19 — Консолидация #1
- Обработано эпизодов: 12
- Новых принципов: 2 (архитектура, тестирование)
- Обновлённых принципов: 1 (миграции)
- Удалённых эпизодов: 3 (дубликаты)
- Новых навыков: 0
- Кандидаты на навык: «миграция БД» (3 повторения)
```

## Подагент-компаньон: режимы работы

### Режим 1: Извлечение (после сессии)

**Триггер:** завершение сессии основного агента (PostToolUse → Stop)

**Вход:** diff текущей сессии, текущее содержимое `.memory/`

**Промпт подагента:**

```
Ты — агент-память. Твоя задача — извлечь значимые события из завершённой сессии и обновить память проекта.

1. Прочитай git diff последней сессии (git log -1 --format=%H | xargs git diff HEAD~1)
2. Прочитай .memory/episodes.jsonl (последние 20 записей для контекста)
3. Прочитай .memory/principles.md

Определи значимые события сессии:
- Архитектурные решения (почему выбрали X, а не Y)
- Обнаруженные баги и их причины
- Повторяющиеся паттерны (делали похожее раньше?)
- Инсайты (что узнали нового о проекте/стеке)
- Откаты (что пробовали и отменили, почему)

Для каждого события добавь строку в episodes.jsonl в формате:
{"ts": "<ISO8601>", "session": "<hash>", "type": "<тип>", "summary": "<одно предложение>", "tags": ["<1-3 тега>"]}

Если обнаружил повторяющийся паттерн (3+ похожих эпизода) — обнови principles.md.
Если увидел кандидата на навык — отметь в consolidation-log.md.

НЕ записывай:
- Рутинные операции (стандартные коммиты, запуск тестов без проблем)
- Информацию, уже присутствующую в episodes.jsonl
- Детали реализации (они в коде и Git)
```

**Модель:** Claude Haiku (дешёвая, достаточная для извлечения)

**Выход:** обновлённые episodes.jsonl, возможно principles.md

---

### Режим 2: Консолидация (периодическая)

**Триггер:** каждые 10 сессий ИЛИ когда episodes.jsonl > 100 записей ИЛИ ручной запуск

**Промпт подагента:**

```
Ты — агент-архивариус. Твоя задача — консолидировать накопленную память проекта.

1. Прочитай ВСЕ записи из .memory/episodes.jsonl
2. Прочитай .memory/principles.md
3. Прочитай список файлов в .memory/skills/

Выполни:

A) ГРУППИРОВКА: Сгруппируй эпизоды по тегам/темам. Найди кластеры похожих событий.

B) ИЗВЛЕЧЕНИЕ ПРИНЦИПОВ: Для кластеров из 3+ похожих эпизодов — сформулируй правило и добавь в principles.md. Правило = конкретное действие, не пожелание.

C) ОБНАРУЖЕНИЕ НАВЫКОВ: Если одна и та же последовательность действий повторяется 3+ раз — создай файл навыка в .memory/skills/{slug}.md по шаблону (когда применять, предусловия, шаги, проверка успеха, антипаттерны).

D) ОЧИСТКА: Удали из episodes.jsonl:
   - Дубликаты (одно и то же событие записано дважды)
   - Устаревшие записи (противоречат текущему коду или более свежим записям)
   - Эпизоды, полностью покрытые извлечёнными принципами (старше 30 дней)
   Оставь последние 50 записей нетронутыми.

E) ПРОВЕРКА principles.md:
   - Удали принципы, противоречащие текущему коду
   - Объедини дублирующие принципы
   - Убедись, что файл ≤100 строк

F) Запиши в consolidation-log.md: дату, кол-во обработанных эпизодов, новые/обновлённые/удалённые принципы, новые навыки.
```

**Модель:** Claude Sonnet (требуется анализ и синтез)

**Выход:** обновлённые episodes.jsonl, principles.md, возможно новые skills/, consolidation-log.md

---

### Режим 3: Инжекция (перед сессией)

**Триггер:** начало новой сессии основного агента

**Механизм:** Не требует отдельного подагента. Реализуется через CLAUDE.md, который ссылается на файлы памяти:

```markdown
# Память проекта

При начале работы прочитай:
1. .memory/principles.md — извлечённые правила проекта
2. Последние 10 записей из .memory/episodes.jsonl — недавние события
3. Если задача похожа на существующий навык — прочитай соответствующий файл из .memory/skills/
```

Альтернатива (автоматическая): pre-session hook, который подагентом формирует сводку релевантного контекста из памяти и инжектирует в начало сессии.

---

## Реализация через хуки Claude Code

### Вариант A: PostToolUse hook (автоматический)

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Stop",
        "command": "claude -p \"$(cat .memory/prompts/extract.md)\" --model haiku"
      }
    ]
  }
}
```

Файл `.memory/prompts/extract.md` содержит промпт режима 1.

### Вариант B: Ручной запуск через подагент

```bash
# Извлечение после сессии
claude -p "$(cat .memory/prompts/extract.md)" --model haiku

# Консолидация
claude -p "$(cat .memory/prompts/consolidate.md)" --model sonnet

# Проверка состояния памяти
cat .memory/episodes.jsonl | wc -l    # кол-во эпизодов
wc -l .memory/principles.md           # размер принципов
ls .memory/skills/ | wc -l            # кол-во навыков
```

### Вариант C: Периодическая консолидация через cron/loop

```bash
# Каждые 10 сессий или раз в день
claude -p "$(cat .memory/prompts/consolidate.md)" --model sonnet
```

## Фазы внедрения

### Фаза 1: Файловый фундамент (день 1)

**Задачи:**
- [ ] Создать директорию `.memory/` с начальной структурой
- [ ] Создать пустой `episodes.jsonl`
- [ ] Создать `principles.md` с начальными правилами из CLAUDE.md (перенести подходящие)
- [ ] Создать `consolidation-log.md`
- [ ] Создать `skills/` с 1–2 навыками из текущего проекта (миграция, деплой)
- [ ] Добавить в CLAUDE.md ссылки на файлы памяти
- [ ] Добавить `.memory/` в `.gitignore` (опционально — зависит от потребности в версионировании)

**Критерий готовности:** Основной агент при старте сессии читает principles.md и последние эпизоды.

### Фаза 2: Подагент-извлечение (день 2–3)

**Задачи:**
- [ ] Написать промпт извлечения (`.memory/prompts/extract.md`)
- [ ] Настроить PostToolUse hook в settings.json
- [ ] Провести 5 тестовых сессий, проверить качество извлечённых эпизодов
- [ ] Откалибровать: что считать «значимым событием» (слишком много → шум, слишком мало → потеря)

**Критерий готовности:** После каждой сессии в episodes.jsonl появляются 1–5 релевантных записей без дубликатов.

### Фаза 3: Консолидация (день 4–5)

**Задачи:**
- [ ] Написать промпт консолидации (`.memory/prompts/consolidate.md`)
- [ ] Запустить первую консолидацию после 20+ эпизодов
- [ ] Проверить: принципы корректны? Навыки полезны? Удалённые эпизоды действительно дубликаты?
- [ ] Настроить триггер (ручной или по cron/loop)

**Критерий готовности:** principles.md содержит 10+ правил, извлечённых из реального опыта. Хотя бы 1 навык создан автоматически.

### Фаза 4: Процедурная память (неделя 2)

**Задачи:**
- [ ] Определить 5–10 повторяющихся процедур в текущем проекте
- [ ] Создать шаблон навыка (`.memory/skills/_template.md`)
- [ ] Настроить подагент на автоматическое предложение навыков при обнаружении паттерна
- [ ] Проверить: агент использует существующие навыки при похожих задачах?

**Критерий готовности:** Агент при обнаружении задачи, совпадающей с навыком, предлагает использовать готовый рецепт.

### Фаза 5: Мульти-агентная координация (неделя 3+)

**Задачи:**
- [ ] Сделать `.memory/` общей для всех параллельных агентов (через shared worktree или symlink)
- [ ] Добавить поле `agent_id` в записи episodes.jsonl
- [ ] Научить подагент-память обнаруживать конфликты между агентами
- [ ] Настроить эскалацию при конфликтах

**Критерий готовности:** Два параллельных агента не дублируют работу; конфликтующие решения обнаруживаются.

## Метрики успеха

| Метрика | Цель | Как измерять |
|---------|------|-------------|
| Повторные ошибки | 0 повторений уже записанных ошибок | grep по episodes.jsonl + ручная проверка |
| Размер principles.md | 20–80 строк (не пустой, не раздутый) | `wc -l .memory/principles.md` |
| Кол-во навыков | 5–15 за первый месяц | `ls .memory/skills/ \| wc -l` |
| Использование навыков | Агент ссылается на навык ≥1 раз/неделю | Ручная проверка сессий |
| Стоимость подагента | <5% от стоимости основного агента | Сравнение токенов Haiku vs Opus |
| Точность эпизодов | >80% записей — действительно значимые | Ручная проверка каждые 2 недели |

---

## Ограничения и риски

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Подагент записывает шум | Высокая | Калибровка промпта + ручная проверка первые 2 недели |
| Дрейф принципов при консолидации | Средняя | Версионирование в Git + diff-review при каждой консолидации |
| episodes.jsonl растёт бесконтрольно | Средняя | Лимит 200 записей + автоудаление при консолидации |
| Навыки устаревают | Низкая | Дата создания в файле + ревизия при консолидации |
| Конфликт записи при параллельных агентах | Средняя (фаза 5) | File lock на episodes.jsonl ИЛИ агент-писатель как единая точка записи |
